## 1. 构建目标

项目的目标是把传统 OCR 工具升级为 OCR 智能体。传统 OCR 通常只负责“上传文件、识别文字、下载结果”，但真实文档处理需要更多环节：

- 判断文档类型。
- 选择合适模型。
- 修复表格、公式、图片和阅读顺序。
- 优化输出排版。
- 让 OCR 结果可问答、可翻译、可整理。
- 在长耗时任务中持续向用户反馈进度。

因此，本项目采用“文档处理流水线 + Web 工作台 + OCR 后 RAG”的设计。

## 2. 第一阶段：六层 OCR 架构

最初设计了六层架构：

1. 文件接入。
2. 文档体检。
3. 策略路由。
4. 多 OCR 引擎解析。
5. LLM/规则纠错。
6. 质量验收与输出。

这个架构让项目从一开始就不是单模型绑定，而是可以接入多个 OCR 引擎。后续增加 Dolphin-v2、RAG、HTML 导出时，也能自然挂接到管线中。

## 3. 第二阶段：后端 Pipeline

后端使用 FastAPI 提供 API，并把 OCR 主逻辑放在 `OCRPipeline` 中。Pipeline 的职责包括：

- 接收 PDF 路径。
- 调用 `PDFAnalyzer` 获取文档特征。
- 根据首选引擎和推荐引擎选择 OCR 适配器。
- 执行 OCR。
- 运行表格、公式、阅读顺序纠错。
- 进行质量评分。
- 清洗 Markdown。
- 返回统一结果。

这样前端不需要理解每个模型的细节，只要调用统一 API 即可。

## 4. 第三阶段：Web UI 工作台

前端最初是 OCR 转换工作台，后来根据使用反馈升级为 OCR 智能体页面。关键设计变化：

- 从“单文件转换”升级到“多文档处理”。
- 从“上传完成进度条”升级到“智能预估 OCR 进度”。
- 从“等待结果”升级到“过程消息窗口”。
- 从“提问入口”改成“对 OCR 后文档提问”。
- 增加默认操作标签，降低用户不知道问什么的门槛。

界面设计遵循产品经理思路：用户首先需要知道系统在做什么、处理到哪一步、失败时哪里出错、成功后下一步可以做什么。

## 5. 第四阶段：科研论文增强

普通 OCR 对科研论文常见问题包括：

- 多栏正文顺序错乱。
- 图片没有被抽取或放回原位置。
- 表格识别不完整。
- 公式被拆碎。
- 标题、摘要、正文、参考文献层级不清楚。

因此项目增加 Dolphin-v2，并把它纳入复杂科研论文路由。对于 `complex_scientific_paper` 或含有图片、表格、公式的学术 PDF，系统会优先尝试 Dolphin-v2。

## 6. 第五阶段：OCR 后 RAG

用户进一步希望“做一个 RAG”，并明确提问应基于 OCR 后文档。于是系统加入：

- OCR Markdown 自动入库。
- 文档切块。
- 本地轻量检索。
- LLM Provider 可选回答。
- 无 LLM 时的抽取式 fallback。
- 多文档知识库列表。
- 引导标签和常用任务。

这一步让项目从“文档转换工具”变成“文档知识助手”。

## 7. 第六阶段：输出排版与 HTML

针对 OCR 结果“排版不够像原文、图片没加载、表格不正确”的反馈，增加了 `scripts/export_html.py`。它尝试：

- 渲染 PDF 页面图。
- 抽取图片区域。
- 抽取表格截图。
- 将 OCR Markdown 重排为学术风格 HTML。
- 统一正文字号、行距、标题层级和阅读宽度。

该方案不是完美坐标级还原，但能显著提升 OCR 输出的可读性和展示性。

## 9. 关键决策总结

- 用多引擎路由，而不是绑定单一 OCR。
- OCR 不强制依赖 LLM，但 LLM 可作为增强层。
- 进度显示必须反映后端真实任务，而不是只显示上传进度。
- 科研论文要单独处理，因为它对表格、图片、公式和阅读顺序要求更高。
- OCR 结果应直接进入知识库，否则用户还要自己复制粘贴。
- 项目文档和交接材料是项目交付的一部分，不是附属品。
